01
Workflow · Sequential

Prompt Chaining

한 작업을 여러 단계로 쪼개 직렬로 LLM을 호출한다. 각 단계는 이전 단계 출력을 입력으로 받고, 호출 자체는 독립이다. 단계 사이의 검증 게이트로 망가진 출력을 조기에 끊는다.

chain() 사용 예

python 
chain("친환경 패션", prompts=[
    "3문장 개요를 만든다.",
    "톤(미니멀, 진정성)에 맞게 다듬어라.",
    "200자 본문으로 확장하라.",
])
  • 검증 게이트 — if "<error>" in out로 중단. system 프롬프트에 "문제 있으면 <error>로 답하라" 지시가 있어야 트리거.
  • 독립 호출 — 이전 turn이 messages로 누적되지 않음. 의도된 stateless.
  • 쓸 때 — 단계가 명확히 분리되고 중간 산출물 검증이 중요할 때. 단계가 서로 영향을 줘야 하면 multi-turn으로.
체이닝의 핵심은 단계 분리가 아니라 각 단계 출력을 다음 단계의 입력 스키마에 맞게 좁히는 것이다. 단계 사이 형식이 흔들리면 게이트가 무의미해진다.

Tool Choice — 모델이 도구를 고른다

tools는 메뉴일 뿐, 실행은 모델 판단이다. 매 turn마다 도구를 부를지(tool_use) 텍스트로 답할지(end_turn)를 모델이 결정하고, 어떤 도구를 부를지는 description이 좌우한다.

  • 병렬 차단 — {"type": "auto", "disable_parallel_tool_use": true}로 한 응답에 도구 1개만, 순차 추론을 강제.
  • 도구 선택 정확도의 절반은 description — 입력의 형식·단위·범위와 사용 시점, 다른 도구와의 경계를 1~2줄로 명확히.

Prompting · Structure

XML 태그로 입력 감싸기

Claude에서 가장 일반적인 입력 구조화 방식. 지시/데이터 경계를 명확히 하고, prompt injection 1차 방어가 된다. 예약 태그는 없고 이름은 자유 — 의미에 맞게만 쓰면 된다.

입력 감싸기 패턴

python 
llm_call(f"{p}\n\n<input>\n{out}\n</input>")
관습 태그
용도
<input>
일반 입력
<document> / <documents><doc>…</doc></documents>
긴 문서, 다중 문서
<context>
배경 정보
<example>
few-shot 예시
<rubric> / <criteria>
평가 기준
<answer>
최종 답 위치 (모델이 자동으로 감싸기도 함)
  • 주의 — <thinking>: 시키지 않아도 Claude가 사고 과정을 이 태그에 자발적으로 넣는 경향. 다른 용도로 쓰지 말 것.
  • 일관성 — 한 프롬프트 안에서 같은 의미는 같은 태그.
  • 대안 — 코드면 트리플 백틱, 구조화 데이터면 JSON. 마크다운 ###은 경계가 약해 비권장.
  • 특수 기능과 혼동 금지 — extended thinking, prompt caching은 API 파라미터지 XML 태그가 아님.
사용자 입력을 감쌀 땐 입력 안에 같은 태그가 들어있을 가능성을 고려한다. 필요하면 <user_input id="abc123">…</user_input>처럼 구분자를 더해 경계를 강화.
